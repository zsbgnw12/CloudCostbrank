云管权限体系

  一、两道门

  每个请求过两道门：
  1. 模块开关（site-wide）：api_module_permissions 表里 16 个模块，admin 可 PATCH 关停任一模块，一关全员看不见。
  2. 角色守卫（per-user）：按 users.roles 或 Casdoor token 里的 roles 判。cloud_admin 是超级角色，默认满足任何
  require_roles。

  第三层是数据范围：通过 user_cloud_account_grants + API Key 的 allowed_cloud_account_ids 控制具体能看哪些云账号的数据。

  四、三种鉴权方式（都会映射到以上角色）

  1. Cookie cc_access_token：浏览器 Casdoor 登录回调写入的 HttpOnly cookie
  2. Authorization: Bearer <JWT>：云管自签 HS256（本系统签的）或 Casdoor 直签（外部系统直传）
  3. X-API-Key: cck_xxx：由 admin 在 /api/api-keys 建；可额外窄化 allowed_modules + allowed_cloud_account_ids；API Key
  没显式白名单且 owner 是 admin 时 = 继承 owner 全量

  五、典型组合示例

  - 运维值班：cloud_ops + 授权 Azure + AWS 两个账号 → 看这两家的 dashboard 和可触发同步，看不到账单
  - 财务结账：cloud_finance + 授权所有账号 → 看所有月账单、可调账/确认/生成；看不到部署/同步/用户管理
  - 三方系统拉费用：cloud_admin 建 API Key，勾 allowed_modules=["billing","dashboard"] + allowed_cloud_account_ids=[3] →
   只能调这 2 个模块、只看 account 3 的数据
  - 老板只看：cloud_viewer + 授权全部账号 → 只能看 dashboard，任何写操作都 403
